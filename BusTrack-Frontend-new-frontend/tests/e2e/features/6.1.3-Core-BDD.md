# 6.1.3 Core Behavior-Driven Development (BDD)

## Enfoque metodológico

La estrategia Core Behavior-Driven Development (BDD) del proyecto **BusTrack** se orienta a la validación del comportamiento observable de la aplicación desde la perspectiva del usuario y de los servicios expuestos por el backend. Para ello, se adopta **Cucumber** como lenguaje de especificación ejecutable en lenguaje natural y **Cypress** como herramienta de automatización para pruebas de interfaz en un frontend desarrollado con **Vue.js**.

La formulación de escenarios en lenguaje **Gherkin** permite mantener trazabilidad entre las **User Stories (US)**, las **Technical Stories (TS)** y los criterios de aceptación. En consecuencia, cada conjunto de escenarios fue distribuido en archivos independientes dentro de `tests/e2e/features/`, con el fin de preservar la modularidad, la legibilidad y la mantenibilidad de la especificación.

## Distribución de las especificaciones

- `tests/e2e/features/RutasFavoritas.feature`: Gestión de Rutas y Horarios.
- `tests/e2e/features/AdministracionFlotaMonitoreo.feature`: Administración de Flota y Monitoreo.
- `tests/e2e/features/ExperienciaUsuario.feature`: Experiencia del Usuario.
- `tests/e2e/features/LandingPage.feature`: Landing Page.
- `tests/e2e/step-definitions/ejemplo-integracion-backend.js`: ejemplo de integración con el backend mediante `cy.intercept`.

## Criterio de implementación

La separación de escenarios en archivos distintos responde a una práctica recomendable en BDD: un archivo por capacidad funcional principal. Este enfoque facilita la ejecución selectiva de pruebas, simplifica la revisión académica del material y mejora la trazabilidad entre requisitos y automatización.
